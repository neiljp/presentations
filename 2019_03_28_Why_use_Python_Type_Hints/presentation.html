Type Hints in Python 3.6+

Original

1 def activate(x, ctx=None):
2     # 30 lines of code to read

With Type Hints

1 from typing import Optional, Iterable, Dict  # Special types for hinting
2 def activate(x: Iterable[Entry],
3              ctx: Optional[Dict[str, int]]=None) -> bool:
4     # some code...
5     q: List[int] = []
6     # ... more lines

Syntax Notes

  • Describe parameters inline: 'name: Type=value'
  • Arrow (->) before function return type (bool here)
  • Python 3.6 introduced variable types inline (if needed) [PEP 526]

Some simple syntax (PEP 484)

Builtin types (doesn't need typing)

bool, str, int, float, bytes, ...

None

'None' also used for the type of None (not 'NoneType')

Data-structures (generics)

No parameters means 'of anything'

List[bytes] (only bytes!)
Dict[str, int] (only str keys, only int values!)
Set[None]
Tuple[str, int, str], or Tuple[str, ...] (variable length, same type)

Functions

Callable, or Callable[..., int] (any parameters, returns int)
... or eg. Callable[[str, int], None]

Classes

...are just their names :)

What does "Gradual" mean?

Code with and without Type Hints can coexist

HOW?

Code with no Type Hints is implied to have 'Any' type

1 def func1(x):
2     return x * "some string"

is equivalent to:

1 from typing import Any
2 def func2(x: Any) -> Any:
3     return x * "some string"

some_type ↔ Any ↔ another_type

The following usage is likely unintended - but mypy shows no errors!

1 func1("some x") + 1
2 func2("some x") + 1

(unless you use options which require type hints!)

NOTE: Explicitly using 'Any' means you're telling the type-checker:
... "this code is dynamic, just trust me!" (for now)

1. Zulip: REQ [Python 3.4+; was 2.7]

Extract expected HTTP parameters from Django HttpRequests in Views

Usage

  • Apply decorator to a view function (@has_request_variables)
  • Tag function parameters with default value of 'REQ'

Decorator functionality

  • Extract tagged entries from parameter list (defaulting to 'REQ')

Customizability via REQ parameters

  • Explicit validator & converter functions
  • Default values

SUMMARY: Super-useful pattern - completely dynamic

1. Zulip: REQ Type Hints

REQ can be typed in a stub as a function

Slightly simplified request.pyi:

 1 ResultT = TypeVar('ResultT')
 2 
 3 def REQ(whence: Optional[str] = None,
 4      *,
 5      type: Type[ResultT] = Type[None],
 6      converter: Optional[Callable[[str], ResultT]] = None,
 7      default: Union[_NotSpecified, ResultT, None] = Optional[NotSpecified],
 8      validator: Optional[Validator] = None,
 9      str_validator: Optional[Validator] = None,
10      argument_type: Optional[str] = None,
11      aliases: Optional[List[str]] = None) -> ResultT: ...

Type consistency via ResultT (eg. int):

  • Must match the type of the decorated view function parameter [L: 11]
  • default can be unspecified, None, or that type [L: 7]
  • converter method can be None or a callable returning that type [L: 6]

So.. WHY Type Hints?

✔ Type Hints are here to stay (no longer provisional, as of Python 3.7)

Various potential benefits, across Python versions:

✔ Concise inline typing documentation - for users, developers & editors/IDEs!
✔ Testable internally‑consistent inline types - eg. via mypy
✔ Additional tooling to improve refactoring confidence
✔ Apply them only to a subset of your code - Gradual by nature
✔ Could help with migrating from Python 2 → 3

Myths

! No enforced type-checking at runtime - unless you want it
! Don't have to add hints to every variable, even in Type-Hinted functions
! Not going to be a requirement to use them in future Python you write

Caveats

✘ Effort to introduce & maintain (mitigated by tools)
✘ Gains limited by few libraries having Type Hints (or Stubs) (Check PEP 561!)
✘ Gains reduced in code that is highly dynamic!
✘ Finite overhead of extra code, including at runtime (reduced in python 3.7+)

Crazy magic possible, that you didn't think you wanted?
